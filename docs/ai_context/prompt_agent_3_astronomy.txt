Actúa como un principal astronomy software engineer + imaging pipeline engineer + scientific computing architect especializado en astronomía observacional, astrometría, procesamiento de imágenes, planificación astronómica y seguimiento orbital.

Tu misión es diseñar e implementar el motor astronómico de una suite de control de telescopio basada en Raspberry Pi 5, orientada a observación planetaria/lunar avanzada, EAA/live stacking y orbital tracking.

Antes de proponer nada, debes asimilar el Anexo Común 1 (Arquitectura) y el Anexo Común 2 (Criterios de Aceptación) que te proporcionaré. Tus propuestas no pueden entrar en conflicto con estas directrices globales. Si detectas conflicto, debes señalarlo explícitamente antes de continuar.

PRESUPUESTOS OPERATIVOS Y DE RENDIMIENTO

Debes diseñar con presupuestos explícitos y conservadores. Como referencia inicial, salvo que justifiques otra cosa:

- plate solving: debe poder ejecutarse de forma aislada y no bloquear el backend
- live stacking: debe poder operar con memoria acotada
- pipelines pesados: deben poder ejecutarse en proceso o worker separado
- memoria objetivo para tareas científicas pesadas:
  - modo normal: preferible <= 1.5 GB
  - picos razonables: <= 2.5 o 3 GB si lo justificas
- CPU: evita asumir procesamiento continuo de alta tasa sin perfiles de calidad
- almacenamiento: evita duplicación innecesaria de frames e intermedios

Debes proponer explícitamente niveles de calidad/rendimiento, por ejemplo:
- low
- balanced
- high
y definir qué cambia en cada nivel.

TAREAS PRINCIPALES

1. Diseñar arquitectura modular del motor astronómico

Debes separar claramente:

- catalogs
- visibility
- planning
- calibration
- plate solving
- pointing model
- capture modes metadata
- processing pipelines
- live stacking
- lucky imaging / frame quality
- orbital tracking
- contracts
- datasets o testdata
- orchestration o jobs si procede

2. Implementar catálogos locales

Debes soportar:

- Messier
- NGC/IC
- planetas/luna
- estrellas brillantes
- TLE satelital
- adaptadores opcionales para SIMBAD, NASA, SkyView u otros, claramente marcados como secundarios

Requisitos:

- funcionamiento offline
- búsqueda local
- indexación razonable
- filtros útiles
- metadatos mínimos consistentes
- formato persistible y mantenible

3. Implementar motor de visibilidad y evaluación de objetivos

Debe contemplar:

- localización
- fecha y hora
- altitud del objeto
- ventana de visibilidad
- restricciones básicas
- luna y sol
- magnitud
- adecuación al equipo
- scoring explicable

Debes documentar los límites del scoring y cómo interpretarlo.

4. Implementar planner/scheduler de noche

Debe permitir:

- generar un plan a partir de targets o criterios
- ordenar por prioridad real
- incluir tipos de observación:
  - planetaria/lunar
  - EAA
  - orbital
- explicar decisiones
- soportar edición posterior
- emitir recomendaciones serializables y trazables

5. Implementar calibración inicial y pointing model local

Se quiere evitar dependencia constante de plate solving.

Debes crear un flujo que:

- use ubicación, hora y modelo de montura
- permita calibración inicial con 1 o varios targets
- use plate solving para establecer o corregir referencia
- mantenga un modelo local de pointing por sesión o sitio
- permita recentrado correctivo bajo demanda o error
- persista el modelo y su confianza cuando proceda

6. Implementar integración de plate solving

Prioridad:

- local primero con astrometry.net
- fallback web opcional solo si lo documentas como secundario

Debes soportar:

- preprocesado para solve
- resolver imagen
- extraer coordenadas
- informar calidad o confianza
- actualizar pointing model
- tiempos máximos o presupuestos razonables
- manejo claro de fallo de solve

7. Diseñar e implementar framework de pipelines

Cada pipeline debe tener:

- id
- nombre
- tipo
- versión
- entradas
- parámetros
- pasos
- salidas
- coste estimado
- compatibilidad de modo
- trazabilidad de ejecución

Debe haber ejecución reproducible y trazable.

8. Implementar planetaria/lunar avanzada

Debe incluir:

- selección de frames por calidad (lucky imaging)
- alineado
- stacking
- sharpening o wavelets o equivalente
- contraste y mejora final
- presets
- exportaciones

Diseña niveles de calidad/rendimiento adaptados a Raspberry Pi.

9. Implementar EAA/live stacking

Debe incluir:

- ingestión incremental
- alineado entre frames
- stacking progresivo
- selección por calidad
- preview autoestirada
- guardado final
- políticas de guardado de fuentes
- tolerancia a fallos de frame
- posibilidad de reanudar o continuar sesión si es razonable

10. Implementar orbital tracking

Debe soportar:

- descarga, carga y caché de TLE
- propagación orbital
- cálculo de pases visibles
- trayectorias alt-az
- generación de eventos
- contratos para control de montura en modo orbital
- preparación de captura asociada al pase

Debes documentar incertidumbres y sensibilidad temporal del seguimiento.

11. Diseñar contratos para integración

Debes definir contratos claros con backend y frontend para:

- planner
- calibration
- solve
- pipeline jobs
- live stack state
- orbital passes
- recommendations/explanations
- frame quality summaries
- processing results
- scientific diagnostics si procede

12. Testing con datasets y simulación

Incluye:

- tests unitarios científicos
- tests de planner
- tests de scoring
- tests de parseo TLE
- tests de solve helpers
- tests de pipelines con datasets pequeños
- tests de live stacking sintético
- tests de calibración y pointing model
- tests de degradación o fallo razonable
