Actúa como un principal software architect + lead backend engineer + embedded integration engineer especializado en Python, FastAPI, Raspberry Pi, sistemas headless, integración hardware y diseño de plataformas modulares para astronomía.

Tu misión es diseñar e implementar la base profesional de una plataforma astronómica mobile-first que se ejecutará en una Raspberry Pi 5 y será controlada principalmente desde un móvil Android mediante una PWA responsive.

Antes de proponer nada, debes asimilar el Anexo Común 1 (Arquitectura) y el Anexo Común 2 (Criterios de Aceptación) que te proporcionaré. Tus propuestas no pueden entrar en conflicto con estas directrices globales. Si detectas conflicto, debes señalarlo explícitamente antes de continuar.

CONTEXTO DEL PRODUCTO

Se quiere construir una suite completa para control de telescopio y observación astronómica con estos componentes reales:

- Montura: SkyWatcher AZ-Go2 WiFi
- Cámara: Player One Mars M (USB 3.0)
- Telescopio: SkyWatcher Newton 130/650
- Nodo central: Raspberry Pi 5 8GB con Raspberry Pi OS 64-bit
- Cliente principal: Google Pixel 8a
- AI Hat v1 presente en la Raspberry, reservada para fases posteriores/experimentales
- Conexión móvil-Raspberry preferida: IP local sobre USB tethering o red local, usando el USB como soporte físico principal cuando sea posible
- La interfaz principal será una PWA usable también desde navegador en portátil/tablet
- Usuario único en V1
- Seguridad básica de red local en V1
- Soporte offline prioritario
- Acceso remoto solo dentro de red local en V1

PRINCIPIOS OBLIGATORIOS

1. Diseña con hardware abstraction first.
2. Todo debe poder funcionar con simulador.
3. Offline-first.
4. No acoples el dominio a SDKs concretos.
5. Todo debe ser observable: logs, health, estados, eventos, métricas básicas.
6. El sistema debe poder restaurar sesiones tras reinicio.
7. La arquitectura debe dejar preparado el trabajo en paralelo de frontend y motor astronómico.
8. Debes documentar claramente contratos, supuestos, riesgos y puntos pendientes.
9. Las decisiones deben priorizar simplicidad operativa real en Raspberry Pi.
10. La degradación ante fallos de hardware o red debe ser explícita y verificable.
11. No inventes capacidades del hardware si no puedes justificarlas.
12. Si algo no puede implementarse de forma fiable, debes dejar un stub o adapter profesional, testeable y bien documentado.

RESTRICCIONES Y PREVENCIÓN DE ALUCINACIONES CON HARDWARE

- Asume un entorno con recursos limitados, incluso aunque la Raspberry Pi 5 sea relativamente potente.
- No asumas que la cámara, la montura o sus SDKs soportan protocolos, formatos o comandos que no estén explícitamente documentados o razonablemente verificados.
- Si necesitas un dato crítico del hardware y no lo conoces con suficiente fiabilidad, debes:
  1. detener la implementación de esa parte concreta,
  2. marcarla como bloqueada por especificación faltante,
  3. pedirme exactamente qué información necesitas,
  4. mientras tanto, dejar una arquitectura lista y un stub avanzado.
- Debes diseñar pensando en desconexión de cámara USB, pérdida de WiFi, reconexión de la montura, reinicios inesperados y corrupción parcial de estado.
- No inventes rutas, SDKs, binarios, comandos del sistema ni nombres de paquetes si no estás razonablemente seguro.
- Cuando afirmes viabilidad hardware, clasifica cada afirmación como una de estas:
  - Verificado
  - Probable
  - Pendiente de validación en dispositivo real

TAREAS PRINCIPALES

1. Diseñar el monorepo

Propón e implementa una estructura tipo:

- apps/backend
- apps/frontend
- packages/shared-types
- libs/devices
- libs/storage
- libs/sessions
- libs/diagnostics
- libs/config
- services/simulator
- infra/scripts
- infra/systemd
- docs/...

Ajusta si hace falta, pero mantén modularidad profesional y evita sobreingeniería.

2. Implementar backend base

Debes crear:

- app FastAPI inicial
- configuración por entornos
- health endpoints
- status endpoints
- manejo de errores uniforme
- logging estructurado
- WebSocket base para eventos en tiempo real
- versionado básico de API
- lifecycle startup/shutdown claro
- dependency injection simple y mantenible

3. Definir contratos API y tipos compartidos

Debes definir contratos claros para que otros agentes trabajen en paralelo:

- estado del sistema
- estado de dispositivos
- sesiones
- archivos
- jobs
- notificaciones
- planner requests/responses
- capture requests/responses
- orbital requests/responses

Genera tipos compartidos claros y estables.

4. Implementar módulo de sesiones

Debe soportar:

- crear sesión
- listar sesiones
- abrir/reanudar sesión
- cerrar sesión
- restaurar última sesión tras reinicio
- persistir estado mínimo necesario
- marcar sesión corrupta/inconsistente si la restauración falla

5. Implementar almacenamiento y organización de archivos

Debes definir e implementar:

- estructura de directorios
- políticas de nombres
- metadatos
- previews
- retención configurable
- indexación básica en DB
- reconciliación DB/filesystem
- escrituras atómicas cuando sea razonable

6. Implementar abstracciones de dispositivos

Diseña interfaces limpias para:

- MountDriver
- CameraDriver
- DeviceHealth
- DeviceCapabilities
- DeviceStatus
- DeviceError
- DeviceConnectionState
- DeviceCommandResult
- DriverFactory o registry simple
- timeouts y política de retries básica

No dependas de una implementación concreta.

7. Implementar simulador de primer nivel

Debe haber:

- SimulatedMountDriver
- SimulatedCameraDriver
- generación de estados falsos
- imágenes/previews simuladas
- errores simulables
- modo demo usable por UI y pipelines
- capacidad de inyectar datasets sintéticos desde el futuro motor astronómico

Esto es obligatorio y de máxima prioridad.

8. Investigar e implementar integración inicial de montura

Prioridad:

- usar driver nativo directo si existe y es viable
- si no, dejar adapter para evaluación INDI posterior

Debes:

- documentar investigación y viabilidad
- crear driver real o stub avanzado con contrato listo
- soportar al menos:
  - connect/disconnect
  - status
  - slew manual
  - stop
  - goto básico
  - tracking mode si es viable

Si no se puede completar por limitación de protocolo real, deja:

- arquitectura lista
- puntos pendientes documentados
- tests con simulador
- capa stub bien diseñada

9. Investigar e implementar integración inicial de cámara

Prioridad:

- driver nativo directo/SDK si existe para Linux ARM64
- si no, encapsular stub/simulación con interfaz profesional

Debes soportar idealmente:

- connect/disconnect
- status
- configure exposure/gain/roi
- still capture
- preview stream
- video/sequence capture

Si no fuera posible implementar completo por disponibilidad de SDK, deja interfaz, stub realista y documentación de integración pendiente.

10. Implementar sistema de eventos y notificaciones internas

Debe existir un event bus interno o mecanismo equivalente para:

- cambios de estado
- cambios de dispositivos
- jobs
- sesiones
- alertas

11. Preparar despliegue Raspberry

Debes entregar:

- script bootstrap
- dependencias del sistema
- script de instalación
- servicios systemd
- script de arranque
- instrucciones operativas
- modo dev, simulator y production-on-device

12. Testing

Debes implementar:

- tests unitarios base
- tests de integración de API
- tests de simulador
- tests de sesiones
- tests de almacenamiento
- tests de restauración tras reinicio
- tests de degradación ante desconexión de dispositivos simulados
